## 2. Reference: How Agent Teams Work (official)

Source: <https://code.claude.com/docs/en/agent-teams>

- **Lead** = main Claude Code session that creates the team, spawns teammates, coordinates
- **Teammates** = independent Claude Code instances, each with own context window
- **Shared task list** + **mailbox** (message/broadcast) for coordination
- **Display modes**:
  - `in-process` (any terminal, `Shift+Down` to cycle)
  - `tmux` or `iTerm2` split panes (each teammate visible)
  - `auto` (default) = tmux if already inside tmux session, else in-process
- **Subagent definitions reusable**: spawn teammate by subagent type; `tools` allowlist and `model` are honored; body is APPENDED to system prompt (not replacing). ⚠ `skills` and `mcpServers` frontmatter fields NOT applied to teammates — they inherit from project/user settings.
- **Token cost is linear** with teammate count — use only when parallel adds value
- **Sweet spot**: 3-5 teammates, 5-6 tasks each
- **Not supported**: VS Code integrated terminal, Windows Terminal, Ghostty (for split-pane mode)
- **Enable**: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`
- **Force in-process for one session**: `claude --teammate-mode in-process`
- **Team state**: `~/.claude/teams/{team-name}/config.json` (auto-managed, don't edit)
- **Task state**: `~/.claude/tasks/{team-name}/`

## 3. Hooks Relevant to Teams

| Hook | Trigger | Exit 2 semantics |
|---|---|---|
| `TeammateIdle` | Teammate about to go idle | Continue working (send feedback) |
| `TaskCreated` | Task being created | Prevent creation, send feedback |
| `TaskCompleted` | Task marked complete | Prevent completion, send feedback |

## 4. Key Limitations

- No `/resume` or `/rewind` with in-process teammates → lead may ghost-message dead teammates
- Task status can lag (teammates forget to mark complete, blocks dependents)
- One team per session
- No nested teams (teammate cannot spawn its own team)
- Lead is fixed for the session lifetime
- Teammates inherit lead's permission mode at spawn; per-teammate override only after spawn

## 7. V2 Research (2026-04-09, via context7 /websites/code_claude)

### 7.1 Hook stdin payload — confirmed schemas

**TaskCreated / TaskCompleted** stdin fields:
```
session_id, transcript_path, cwd, permission_mode, hook_event_name,
task_id, task_subject, task_description, teammate_name, team_name
```

**TeammateIdle** stdin fields:
```
session_id, transcript_path, cwd, permission_mode, hook_event_name,
teammate_name, team_name
```

**Critical implication for V1 → V2:**

- No `file_paths`, `owned_paths`, `acceptance_criteria`, or `assignee` fields natively.
- V1's plan to detect file conflicts from stdin at TaskCreated time is **impossible without a self-imposed structured format embedded in `task_description`**.
- Hooks must either (a) parse a strict YAML/markdown block inside `task_description`, OR (b) degrade to advisory logging only.

### 7.2 Task storage paths — documented but "auto-managed"

Quoted from docs: *"Agent team configurations and task lists are stored locally. The team configuration, located at `~/.claude/teams/{team-name}/config.json`, stores runtime state like session and tmux pane IDs and **should not be edited manually as it's automatically updated**. Task lists are stored in `~/.claude/tasks/{team-name}/`."*

Also: `CLAUDE_CODE_TASK_LIST_ID=my-project claude` lets callers name a shared task list directory.

**Stability posture:** treat both paths as READ-ONLY best-effort. Never rely on them as single source of truth. Hooks must fall through silently if paths don't exist or schema changes.

### 7.3 `teammateMode: auto` does NOT detect terminal incompatibility

Quoted from docs: *"`auto` intelligently selecting split panes in tmux or iTerm2, or in-process otherwise"* and *"uses split panes if you're already running inside a tmux session"*.

**Gap:** no auto-detection of VS Code integrated terminal, Windows Terminal, Ghostty. If the user is inside `tmux` nested in VS Code, `auto` still tries split panes and may fail.

**V2 mitigation:** runtime probe must include terminal ancestry check (`$TERM_PROGRAM`, `$WT_SESSION`, `$VSCODE_PID`) and downgrade to IN_PROCESS when incompatible.

### 7.4 `TaskCompleted` fires in two cases

Quoted from docs: *"This fires in two situations: when any agent explicitly marks a task as completed through the TaskUpdate tool, or when an agent team teammate finishes its turn with in-progress tasks."*

**Implication:** V2 hooks must distinguish between explicit completion (happy path) and implicit end-of-turn with pending work (signal the lead to reassign or nudge).

### 7.5 Tools allowlist — NOT confirmed as invariant in context7 v2.1.89

The phrase *"SendMessage and task management tools are always available even when `tools` restricts other tools"* appears in the live agent-teams page fetched 2026-04-09 but does NOT appear in the context7 snapshot for v2.1.89. Treat this as **documented behavior as of Claude Code 2.1.32+, verifiable by regression test, NOT a platform invariant**.

**V2 posture:** explicitly declare `SendMessage`, `TaskCreate`, `TaskUpdate`, `TaskList`, `TaskGet` in every teammate agent's `tools:` frontmatter — belt and braces.

### 7.6 Subagent spawn control (`Agent(type)` syntax)

New finding from context7: the `tools` field supports `Agent(worker, researcher)` syntax to restrict which subagent types a lead can spawn, and `Agent` alone (no parens) allows any. `permissions.deny` can block specific subagents globally.

**V2 use:** leads in Phase B can declare `Agent(reviewer-security, reviewer-perf, ...)` in their frontmatter to tighten the blast radius.

### 7.7 Task payload contract (imposed by V2, not by Claude Code)

Since stdin only carries `task_subject` + `task_description`, V2 encodes all structured metadata inside `task_description` using a machine-readable block that the shared `team-mode.md` mandates:

```
<!-- task-contract v1 -->
scope: src/auth/
owned_paths:
  - src/auth/jwt.go
  - src/auth/jwt_test.go
forbidden_paths:
  - src/auth/session.go
acceptance_criteria:
  - all tests pass
  - no clippy warnings
output_format: unified diff
assignee: reviewer-security
depends_on: []
<!-- /task-contract -->

Followed by natural-language details for the teammate.
```

Hooks parse the block with `awk`/`grep`, fall back to advisory mode if absent.
